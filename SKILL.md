# GPU 训练性能优化

## 目标与边界

- 处理吞吐、显存、数据等待、计算路径、编译和扩展效率；批量变化只测量工程收益、更新次数和预计墙钟，不自动改变学习率或训练目标。
- 学习率、预热、调度、增强、软标签和正则化属于训练配方；同时要求速度和指标时，先比较“保持当前批量”和“吞吐拐点批量”，再把值得验证的批量作为独立配方候选。
- 普通复现或按项目既有命令启动训练时，不强制执行完整性能流程。
- 项目的 `AGENTS.md`、实验协议、环境和产物规范优先；不得写死 Conda 环境或个人数据路径。
- 优化必须同时比较速度、显存和目标指标，不能用改变实验定义换取未注明的“提速”。

## 工作顺序

1. 读取项目规则、已有性能记录和实验协议，确认可比指标、允许改动、结果容差和用户更重视速度还是保持原配方。目标主要是提高模型指标时，不在本技能内扩展配方搜索。
2. 读取 [基线与定位](references/baseline-and-profiling.md)，记录可复现基线并判断瓶颈。
3. 涉及批量选择或训练成本预测时读取 [批量与成本评估](references/batch-size-and-cost.md)，不要用单个批次外推完整训练。
4. 视觉卷积网络任务读取 [视觉卷积网络优化](references/vision-cnn-optimization.md)，包括但不限于 ResNet、CIFAR 和 ImageNet；需要选择外部实现时再读 [视觉工程证据](references/vision-engineering-evidence.md)。
   用户要求解释术语或理解训练流程时读取 [视觉优化概念](references/vision-cnn-concepts.md)。
5. 其他任务只读取 [优化手册](references/optimization-playbook.md) 中与已识别瓶颈对应的部分。
6. 按瓶颈和成熟先验排序候选，先确认主效应，再以当前最佳配置为锚点逐项加入；只测试有明确机制依据的组合，不做全排列。
7. 读取 [验收与报告](references/acceptance-and-reporting.md)，决定接受、拒绝或作为独立实验保留。

## 默认与深度模式

默认模式只做不阻塞正式训练的检查：复用已有项目记录，排查资源争用、逐样本 `.item()`、重复解码、重复搬运、频繁日志和明显循环浪费，完成冷启动冒烟与热缓存短基准；只有用户关心批量收益时才扫描少量批量。按证据给出一套最推荐的工程组合，并分别报告已测单项主效应、相对当前最佳配置的增量效果及组合总效果。报告当前批量与吞吐拐点的预计收益、风险和回本次数，由用户决定是否进入配方验证。

深度模式只在用户明确要求、训练会重复或短基准显示存在显著机会时进入，包括 Profiler、代码向量化、图断裂、动态形状、专用数据后端和多卡扩展。深度实验使用独立资源，不阻塞当前正式训练。

## 快速入口

- 只采集运行环境和 GPU 基线：执行 `scripts/capture-training-baseline.sh [采样秒数]`。
- 视觉卷积网络，包括 ResNet、CIFAR 和 ImageNet：读取视觉卷积网络优化，按已定位的瓶颈选择少量候选。
- GPU 利用率低或周期性停顿：重点检查数据管线、同步、日志、评估和检查点。
- 显存不足：重点检查批量、激活、优化器状态和分片；不要默认启用多卡。
- 评估批量：只输出当前批量和吞吐拐点，不把最大批量或固定步最高吞吐直接推荐为训练配置。
- 多卡扩展差：比较单卡与多卡吞吐，核对每卡批量、通信等待和数据不均衡。

## 验收

- 高风险代码优化应在具备基线提交的 Git 项目中隔离；小型配置实验可沿用项目当前流程。
- 未取得可比较基线、目标指标或日志路径时，不宣称优化有效。
- 关键模型、数据、批量、精度和测量窗口保持一致即可；允许显卡和调度噪声带来小幅差异。收益接近噪声时再重复，不要求机械换卡复测。
- 最终报告必须给出基线命令、改动、吞吐差异、显存差异、指标差异、产物路径和适用限制。
