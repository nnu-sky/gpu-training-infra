# GPU 训练优化手册

只使用与已识别瓶颈相关的部分，并保留可关闭的配置入口。

## 精度与计算

- 在硬件与算子支持的前提下分别测试 `bf16` 和 `fp16`，按吞吐、显存、数值稳定性与目标指标选择，不按显卡型号预设答案。
- 仅在框架和算子链明确支持时测试 `fp8`；小型模型或普通卷积任务不默认引入它。
- 稳定形状的 PyTorch 训练可测试 `torch.compile`，分别报告编译开销与稳定吞吐，并记录图断裂和显存变化。
- Transformer 类模型检查 PyTorch 缩放点积注意力实际后端；只有注意力确为瓶颈时才增加 FlashAttention 等依赖。

## 数据管线

- 逐档测试 `num_workers`，同时检查 CPU、存储和 GPU 空闲时间，不以工作进程越多越好。
- 使用页锁定内存时可配合非阻塞传输；长期工作进程和预取必须用实际吞吐验证。
- 高开销且科学上允许复用的解码、几何或特征预处理可缓存到项目批准的数据盘。
- 避免从慢速家目录或网络路径持续读取高吞吐训练数据。

## 同步、日志与保存

- 降低逐步 `.item()`、转 CPU、显式同步、全量指标归约和详细日志的频率。
- 速度实验中减少不必要的评估和大检查点写入，但不得破坏恢复要求。
- 将正确性调试日志与正式吞吐测试分开。

## 批量与显存

- 逐步增加每卡批量，直到吞吐不再改善或显存、稳定性、指标不可接受。
- 需要固定有效批量而显存不足时使用梯度累积，并在比较中保持批量口径一致。
- 激活检查点以计算换显存，通常会减慢单步；只在显存是实际瓶颈时使用。

## 多 GPU

- 普通可容纳模型优先测试 DDP；小模型可能因通信而比单卡更差。
- 只有模型、优化器或激活状态无法合理容纳时，再考虑 FSDP 或 ZeRO 类分片。
- 记录卡数、每卡批量、有效批量、单卡吞吐、多卡吞吐和扩展效率。
- 多卡测试必须检查数据分片、进程到 GPU 的映射、通信端口和不均衡等待。

## 按任务类型

- ResNet、CIFAR 和 ImageNet：读取 `vision-cnn-optimization.md`，不要只使用此处的通用建议。
- 三维、扩散和渲染：分开定位预处理、渲染、数据传输和核心算子，并审查可复用缓存。
- Transformer 与多模态：优先精度、注意力后端、序列分桶、填充浪费、融合算子和多卡策略。
