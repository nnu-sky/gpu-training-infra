# 基线与瓶颈定位

## 启动前

1. 读取项目现有说明、规则、实验协议、当前计划和决策记录；存在 `AGENTS.md` 时一并遵守。缺少这些文件不构成阻塞，也不要仅为性能分析创建管理文档；改从训练入口、配置和已有日志提取约束并明确假设。
2. 记录代码版本、环境、数据集或固定子集、模型、种子和输出目录。
3. 使用项目声明的环境；不要假定环境名或个人目录。
4. 记录设备占用、CPU、内存、存储和现有训练造成的资源竞争；不要把受争用的结果当作候选自身性能。

## 最小可比较基线

至少记录：

- 完整命令和代码版本；
- 模型、数据、种子和核心指标；
- 精度、每卡批量、梯度累积、卡数和有效批量；
- 数据加载工作进程、页锁定内存、常驻工作进程和预取设置；
- 预热后的步耗时或样本吞吐、GPU 利用率和峰值显存；
- 数据等待时间、GPU 计算时间和完整单步时间，三者使用一致的采样窗口；
- 编译、缓存和首步开销是否计入；
- CPU、存储或数据读取是否出现明显等待。

快速探索可使用固定的小批次和固定步数，但前后测试必须保持相同口径。不要把首步编译或缓存时间当作稳定吞吐。
正式接受优化时还要记录完整训练时长或达到目标验证指标所需时间；每秒样本数提高不等于更快达到目标精度。

训练循环内部计时优先使用已有指标。新增测量时，完整单步使用单调墙钟，GPU 计算区间使用 CUDA Event；不要为了逐步计时在每一步加入全局同步。PyTorch Profiler 只在有限的等待、预热和活动步内启用。

## 只读采样

训练运行时在另一个终端执行：

从本技能的目录执行 `scripts/capture-training-baseline.sh [采样秒数]`，不要假定技能位于固定的个人目录。未指定时可从短采样开始；单步较慢、利用率波动大或冷启动较长时延长，直到观察窗口能代表稳定阶段。

脚本输出主机、GPU 型号、显存、计算进程和按秒利用率。需要算子级定位时，再使用项目已有的 PyTorch Profiler、Nsight Systems 或框架分析工具；先限定少量预热与采样步数，避免剖析器本身主导性能。

## 瓶颈判断

| 现象 | 优先核查 |
| --- | --- |
| GPU 利用率低且数据等待高 | 解码、变换、存储、缓存、工作进程、主机到设备传输 |
| GPU 利用率高但单步慢 | 精度、算子、形状、注意力、批量、编译机会 |
| 显存高且吞吐低 | 激活、批量、优化器状态、检查点和分片 |
| 多卡增益小 | 每卡批量、通信占比、数据不均、同步和启动方式 |
| 周期性停顿 | 评估、保存、日志、`.item()`、显式同步和工作进程重启 |

先形成一个可证伪的瓶颈判断，再选择优化。若证据不足，延长只读采样或缩小剖析范围，不同时改多个方向。
