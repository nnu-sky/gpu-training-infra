# ResNet、CIFAR 与 ImageNet 工程优化

只保留可迁移的工程流程。具体项目的吞吐、批量、精度组合和拒绝结论写入项目报告，不回填为通用默认。

## 比较边界

- 纯工程比较保持模型、数据、增强意图、损失、优化器、学习率计划、种子和有效批量不变。
- 工作进程、缓存状态、显卡与系统负载允许存在小幅噪声，但必须记录；收益接近噪声时再复测。
- 改变有效批量、学习率、预热、增强或正则化属于新训练配方，转用 `vision-classification-tuning`。
- 不预设 BF16、FP16、`channels_last`、`torch.compile`、更多数据进程或多卡一定有效；组合效果也必须实测。

## 默认流程

### 1. 建立锚点

- 记录当前命令、代码版本、环境、GPU、数据位置、模型、批量、精度和增强配置。
- 分开记录冷启动、稳定训练、数据等待、GPU 计算、每秒样本数、峰值显存、每轮墙钟和项目指标。
- 先复用已有可靠日志；缺少可比锚点时再运行短基准。
- 短基准至少覆盖预热和多个稳定步骤，不能只用一个批次外推完整训练。

### 2. 检查项目代码与输入管线

优先排查无需改变训练配方的明显浪费：

- 每个样本重复读取、解码或预处理；
- PIL、数组和张量之间不必要的往返转换；
- 重复搬运到 GPU，或本可批量执行却逐样本循环；
- 每步 `.item()`、转 CPU、显式同步或全量指标归约；
- 过密日志、验证和检查点写入；
- 每轮重新创建数据工作进程；
- 损失或辅助统计只为日志使用，却在训练主路径反复同步。

只修改已被计时或代码检查证明的重要位置。代码向量化必须用数值与梯度测试确认等价，检查点键名和恢复流程也必须保持兼容。

### 3. 选择候选与组合

不要把所有开关逐项穷举后再做全排列。先根据瓶颈选择候选族：数据等待高时优先数据加载与传输；GPU 计算占主导时优先精度、内存布局与编译；同步或循环占主导时先修复代码热点。没有机制依据的候选不运行。

使用顺序增量验证：

1. 建立原始锚点 `A`。
2. 测试预计主效应最大的单项 `B`；无收益则关闭。
3. `B` 有效后，以 `A+B` 为新锚点，再测试互补候选 `C`，报告 `C` 相对当前最佳配置的增量，而不是只报告组合总提升。
4. 对已知可能存在交互的组合，可先比较 `A` 与 `A+B+C`；组合有效后只做最小消融，分别去掉 `B` 或 `C`，不扩展为全部排列。
5. 增量收益接近噪声、完整墙钟不再缩短、指标越界或维护成本超过收益时停止。

最终推荐可以是一套组合，但必须区分：单项主效应、加入当前最佳配置后的增量效果、组合总效果，以及未单独验证的组成项。成熟先验只决定测试顺序，不能替代本项目测量。

### 4. 调整数据加载

- 小范围扫描 `num_workers`，同时观察 CPU、存储、数据等待和 GPU 空闲；工作进程不是越多越好。
- 分别测试页锁定内存、非阻塞传输、常驻工作进程和预取，不把整组开关一次性归因给单项。
- CIFAR 的标准实现通常在初始化后驻留于内存，先检查增强、转换和同步开销，不默认引入专用数据后端。
- ImageNet 类逐文件 JPEG 数据先比较本地盘、热缓存和冷缓存，再判断是否受读取、解码或增强限制。
- 只有标准 DataLoader 仍是明确瓶颈时，才评估 TorchVision v2、格式转换与缓存、FFCV 或 NVIDIA DALI；更换后端后核对样本、标签、插值、归一化和增强分布。

### 5. 测试计算路径

- 在硬件和算子支持的前提下分别比较 FP32、BF16 和 FP16；FP16 通常需要梯度缩放。
- 同时检查吞吐、显存、非有限损失、溢出、验证指标和检查点恢复。
- 在混合精度有效后再测试模型与输入的 `channels_last`，并检查是否存在反复格式转换。
- 固定输入尺寸时可测试 cuDNN 自动调优，但要注明可复现性变化。
- 组合开关需单独测量；单项有效不等于组合一定有效。

### 6. 测试 `torch.compile`

- 先检查形状稳定性、图断裂、动态控制流、编译器依赖和缓存写权限。
- 默认只编译训练主路径；验证很短时保持普通模式，避免为不同训练状态和输入形状额外编译。
- 分开报告首次编译、预热、稳定吞吐和完整墙钟。至少跨越两个训练周期，观察尾批和训练／验证切换是否触发重编译。
- 尾批形状不同可能生成新图；短基准可在固定形状窗口停止，但完整任务估算必须计入首轮编译和尾批成本。
- 编译包装不得改变公开模型对象、检查点键名或加载方式。稳定吞吐没有收益时停止，不继续堆叠编译模式。

固定步变快但完整训练变慢时，依次核对：首轮编译额外时间、编译图数量、尾批单步时间、每轮数据重排与工作进程行为、周期性日志／保存／评价，以及完整周期的中位墙钟。分别计时完整批量和尾批；若尾批或新形状占用主要时间，测试丢弃尾批、填充固定形状或保持普通模式的独立候选。无法定位时使用短 Profiler 窗口，不把学习率问题解释为工程墙钟问题。

### 7. 评估批量

- 先在原批量完成工程对照，再按几何级数扫描每卡批量，直到吞吐进入平台、显存风险上升或稳定性下降。
- 输出“当前批量”和“吞吐拐点批量”，不要把最大可容纳批量或固定步最高吞吐直接推荐为训练配置。
- 估算完整训练时同时计算每轮更新次数、每轮墙钟、总墙钟和首次编译摊销，不能只看单步速度。
- 批量改变后，把学习率、预热和调度联动交给 `vision-classification-tuning`；保留原批量方案作为可比较锚点。

### 8. 深度模式与多卡

- 只有默认流程仍有显著未解释时间时，才使用受控窗口的 PyTorch Profiler 定位算子、同步、数据或通信瓶颈。
- 深度代码优化可处理热点循环、重复计算、图断裂和可向量化路径；避免为小收益引入大型包装层。
- 模型可单卡容纳时先完成单卡优化，再测试 DDP；不使用 `DataParallel`。
- 分别报告固定全局批量的强扩展和固定每卡批量的最大吞吐。后者改变训练配方。
- 记录拓扑、NUMA、每卡批量、总工作进程、通信等待和扩展效率；多卡更慢时保留单卡。
- FSDP、ZeRO 或激活检查点只在模型或状态无法合理容纳时考虑，不作为小型卷积网络默认路径。

## 验收与记录

最终报告至少包括：

- 原始锚点与可复现命令；
- 每个实际改动、作用机制和单独测量结果；
- 最佳同批量工程配置；
- 批量吞吐拐点及其配方风险；
- 吞吐、显存、完整墙钟和项目指标变化；
- 被拒绝的方法及原因；
- 代码、日志、报告和检查点位置。

项目结论留在项目记录中。技能只保留上述判断流程和外部资料入口。
