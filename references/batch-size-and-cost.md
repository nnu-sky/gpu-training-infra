# 批量与训练成本评估

## 目标与边界

本流程只寻找吞吐拐点、估算训练成本并帮助用户判断是否值得改变批量。它不证明准确率，也不自动改变学习率、预热、调度或权重衰减。

只向用户比较两个方案：保持当前批量，以及采用吞吐拐点批量。最大批量和固定步最高吞吐只作为内部诊断，不作为默认训练候选。

## 测量方法

1. 固定模型、数据路径、精度、增强、工作进程、设备数量和随机种子。
2. 从当前批量开始逐级增加少量候选；出现显存不足、吞吐下降或边际收益很小时停止。
3. 每个候选先排除初始化、数据工作进程启动和编译等冷启动，再测量稳定窗口。可从预热 20～50 步、测量 50～100 步开始；单步很慢、周期很短、动态形状明显或编译开销较大时按稳定性调整，不把这些数字当作硬门槛。收益接近噪声时再重复热缓存测量。
4. 单个批次不能代表稳定吞吐；首次数据读取、CUDA 初始化、算法选择、异步计时和编译会污染结果。
5. 当验证、工作进程启动、尾批、保存或编译占比明显时，再跑一个或两个热周期校正端到端估计；不要机械要求完整训练。
6. 固定步快但热周期慢时，分别测量完整批量、尾批和周期边界。大批量减少步数后，单个尾批、第二种编译形状、数据重排和周期固定开销的占比会显著放大；原因未定位前不进入完整配方训练。

把“吞吐拐点”定义为继续增大批量后端到端吞吐不再明显增加的最小合理批量。不要用填满显存代替拐点判断。

## 成本计算

设训练样本数为 `N`，批量为 `B`，稳定吞吐为 `Q`，总周期数为 `E`：

```text
每周期更新次数 = ceil(N / B)
理想训练段周期时间 ≈ N / Q
预计训练段总时间 ≈ E × N / Q
预计完整时间 ≈ 训练段 + 验证 + 保存 + 启动／编译固定成本
预计加速倍数 = 当前方案完整时间 / 候选方案完整时间
每次训练节省 = 当前方案完整时间 - 候选方案完整时间
回本训练次数 = 额外调优与验证时间 / 每次训练节省
```

同时报告估计区间而非伪精确值。若只有固定步数据，明确说明验证、保存、数据冷启动和编译未完全计入。

## 批量变更所需信息

用户选择改变批量后，为后续配方验证保留：

- 当前批量、候选批量和倍率；
- 每周期与总更新次数变化；
- 固定步与热周期吞吐、显存、数据等待和尾批形状；
- 预计完整墙钟、节省时间和回本次数；
- 当前学习率、预热、调度、权重衰减和优化器；
- 用户目标是最大化吞吐、达到目标精度的时间，还是保持最终精度。

保持当前学习率不变作为锚点。平方根缩放、线性缩放和预热只作为待验证候选，不能在工程报告中写成自动答案。

## 用户决策

- 收益很小、只训练一次或验证成本高：建议保持当前批量，继续使用其他已通过的工程优化。
- 收益明显且任务会重复：建议对拐点批量做最小联合配方验证。
- 固定步变快但热周期没有缩短：拒绝批量变化，记录输入、验证或固定开销抵消了计算收益。
