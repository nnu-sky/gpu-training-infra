# GPU Training Infra

> 让 Codex 不只会启动训练，还能读懂训练工程、找到主要瓶颈、验证优化，并交付一套真正适合当前项目的推荐配置。

## 为什么需要它

很多训练项目在代码跑通后就开始长时间训练，主要精力放在模型、损失和最终指标上。数据加载、CPU 与 GPU 等待、逐样本循环、计算精度、显存布局和编译等工程问题往往没有被系统检查，昂贵的 GPU 因此没有得到充分利用。

工程优化又横跨 PyTorch、CUDA、数据管线和硬件特性。不同设置还会相互影响，单看显卡占用或瞬时吞吐，很难判断真正的瓶颈和最合适的组合。

`gpu-training-infra` 让 Codex 从代码和运行现象出发，定位主要瓶颈，验证少量高价值改动，并用统一结果说明哪些设置值得保留。

## 真实项目结果

> [!TIP]
> **CIDER：稳定吞吐提升 3.49 倍，峰值显存降低 76%**
>
> CIFAR-10、ResNet18、批量 512：稳定吞吐从 **1,495** 提升到 **5,216 样本/秒**。

| 阶段 | 核心改进 | 稳定吞吐 | 相对原始 | 峰值显存 |
| --- | --- | ---: | ---: | ---: |
| 原始基线 | FP32＋逐样本原型更新 | 1,495 样本/秒 | 1.00 倍 | 13.114 GiB |
| 第一步 | 批量张量聚合替代逐样本循环 | 4,152 样本/秒 | 2.78 倍 | 13.112 GiB |
| 第二步 | 常驻进程＋异步传输＋梯度置空 | 4,161 样本/秒 | 2.78 倍 | 13.114 GiB |
| 第三步 | BF16 自动混合精度 | 4,764 样本/秒 | 3.19 倍 | 2.670 GiB |
| 第四步 | `channels_last` 内存排列 | 4,982 样本/秒 | 3.33 倍 | 2.921 GiB |
| 第五步 | `torch.compile` 编译训练图 | **5,216 样本/秒** | **3.49 倍** | **3.150 GiB** |

最终稳定吞吐提高约 **249%**，峰值显存降低约 **76%**。按稳定吞吐估算，500 轮训练计算段由约 **4.64 小时**缩短至约 **1.33 小时**。同一优化思路在 CIFAR-100、ResNet34 上也得到复现：吞吐从 **1,297** 提升到 **5,119 样本/秒**，达到 **3.95 倍**。

> [!TIP]
> **PALM：稳定吞吐提升 3.92 倍，峰值显存降低 44%**
>
> CIFAR-100、ResNet34、批量 512：稳定吞吐从 **2,510** 提升到 **9,828 样本/秒**。

| 阶段 | 核心改进 | 稳定吞吐 | 相对原始 | 峰值显存 |
| --- | --- | ---: | ---: | ---: |
| 原始基线 | FP16＋2 个数据工作进程 | 2,510 样本/秒 | 1.00 倍 | 7.626 GiB |
| 第一步 | 数据进程由 2 个增至 8 个 | 5,267 样本/秒 | 2.10 倍 | 7.626 GiB |
| 第二步 | BF16 自动混合精度 | 5,173 样本/秒 | 2.06 倍 | 4.234 GiB |
| 第三步 | BF16＋`channels_last` | 7,851 样本/秒 | 3.13 倍 | 4.236 GiB |
| 第四步 | `torch.compile` 编译模型前向图 | **9,828 样本/秒** | **3.92 倍** | **4.274 GiB** |

最终稳定吞吐提高约 **292%**，峰值显存降低约 **44%**。按稳定吞吐估算，500 轮训练计算段由约 **2.77 小时**缩短至约 **0.71 小时**。

## 它能帮你完成什么

### 找出训练慢在哪里

- 识别数据加载、CPU、存储、GPU 计算、显存和通信瓶颈；
- 检查重复解码、重复搬运、逐步同步、频繁日志和循环低效；
- 分析 GPU 利用率、数据等待、单步时间、吞吐和完整训练耗时；
- 区分首次编译、热缓存、尾批和轮次边界带来的额外成本。

### 选择值得尝试的优化

- 比较 FP32、BF16、FP16、`channels_last` 和 `torch.compile`；
- 调整数据工作进程、页锁定内存、预取、常驻进程和异步传输；
- 测量批量大小与吞吐、显存、更新次数和预计训练成本的关系；
- 判断单卡、DDP 或 FSDP 是否适合当前模型；
- 在深度模式下继续分析算子、图断裂、动态形状、向量化和多卡扩展。

### 给出可以直接采用的结论

最终不会只得到一张性能表，而会得到：

- 当前最主要的瓶颈和判断依据；
- 已尝试的优化及其单项收益；
- 不值得保留的尝试和失败原因；
- 推荐的工程配置及组合收益；
- 显存、稳定性和模型指标变化；
- 适用范围、风险以及下一步是否值得继续优化。

## 两种使用模式

### 默认模式

适合日常训练和大多数项目。它会快速检查代码与运行环境，完成冷启动冒烟和热状态短基准，优先处理最明显、最可能产生收益的问题，然后给出一套推荐配置。

默认模式强调快速落地，不要求先完成长时间的全面性能分析。

### 深度模式

适合会反复训练、训练成本较高，或者默认模式发现明显优化空间的项目。它可以继续使用性能分析器，检查算子、编译图、动态形状、代码向量化、专用数据后端和多卡扩展。

深度模式只在额外分析成本可能被后续训练收益覆盖时使用。

## 它如何工作

1. 读取项目说明、配置、训练入口和已有日志；
2. 建立可以复现的速度、显存和目标指标基线；
3. 根据运行现象判断主要瓶颈；
4. 按预期收益和改动成本选择少量候选；
5. 分别测量关键单项和有明确依据的组合；
6. 输出最推荐的工程配置与前后对比。

它不会穷举所有开关，也不会把某个项目的最佳参数复制到另一个项目。批量、吞吐、完整耗时和模型指标会一起分析，再决定哪个方案更适合当前训练任务。

## 快速开始

安装后可以直接对 Codex 说：

```text
使用 $gpu-training-infra 检查这个 PyTorch 项目的训练工程，找出最值得优化的地方，完成短基准，并给出一套推荐配置。
```

也可以针对具体问题：

```text
使用 $gpu-training-infra 比较 BF16、channels_last 和 torch.compile 在这个项目中的实际收益。
```

```text
使用 $gpu-training-infra 测量批量大小、显存和训练时间的关系，告诉我是否值得调整批量。
```

```text
使用 $gpu-training-infra 分析为什么多卡训练没有比单卡更快。
```

## 安装

### 作为插件安装

```bash
codex plugin marketplace add nnu-sky/gpu-training-infra
```

刷新插件市场后安装“GPU 训练性能优化”。这种方式便于后续统一升级。

### 只安装技能

在 Codex 中调用 `$skill-installer`，安装下面的 GitHub 目录：

```text
https://github.com/nnu-sky/gpu-training-infra/tree/main/plugins/gpu-training-infra/skills/gpu-training-infra
```

安装后若没有立即出现，重新启动 Codex。

## 项目兼容性

`AGENTS.md` 不是使用本技能的前置条件：

- 项目已经提供规则或实验协议时，技能会优先遵守；
- 项目没有这些文件时，会从说明、配置、训练入口和日志中提取约束；
- 缺少信息时会明确说明假设；
- 不会仅为了性能分析要求项目创建额外管理文档。

## 安全与验证

- 仓库不包含服务器地址、账号、密钥、令牌、个人路径或私有项目配置；
- 每项优化都应在统一测量口径下比较；
- 性能收益会与显存、稳定性和项目目标指标一起报告；
- 短基准用于筛选候选，重要结论仍需在有代表性的训练窗口中确认；
- 修改训练代码或执行项目脚本前，应先检查其内容和适用范围。
